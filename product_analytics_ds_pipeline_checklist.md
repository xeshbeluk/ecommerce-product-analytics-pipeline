## 0. Data Strategy

### Primary Dataset: Olist Brazilian E-Commerce (Kaggle)
**Why:** Real transaction data with customers, orders, payments, reviews, products, sellers, and timestamps. Sufficient for LTV, churn, segmentation, and forecasting out of the box.

**Download:** https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce  
**Files you need:**
- [x] `olist_orders_dataset.csv` — order lifecycle and timestamps
- [x] `olist_order_items_dataset.csv` — line items and prices
- [x] `olist_order_payments_dataset.csv` — payment values
- [x] `olist_order_reviews_dataset.csv` — satisfaction scores
- [x] `olist_customers_dataset.csv` — customer IDs and location
- [x] `olist_products_dataset.csv` — product categories

### Synthetic Event Layer (Generated by You)
Olist has no session or clickstream data. You will generate a synthetic event layer on top of Olist customer IDs to support A/B testing and funnel analysis. **This is intentional and worth explaining in interviews — you understood the data gap and filled it deliberately.**

**What to generate (Python script in repo):**
- [x] User sessions: `user_id`, `session_id`, `timestamp`, `channel` (organic/paid/email/referral)
- [x] Funnel events: `page_view` → `product_view` → `add_to_cart` → `checkout` → `purchase`
- [x] A/B assignment table: `user_id`, `experiment_id`, `variant` (control/treatment), `assignment_timestamp`
- [x] Experiment definition: 2 experiments minimum — e.g., checkout flow redesign (conversion rate primary metric) and email discount campaign (revenue per user primary metric)
- [x] Bake in a real treatment effect for one experiment (e.g., +3% conversion lift in treatment) and no effect in the other (for a null result — realistic and shows statistical maturity)

**Tools:** Python `Faker`, `numpy`, `pandas`. Seed your RNG for reproducibility.

## 8. Analytics Module 2 — A/B Testing

**This is your most important module for interviews. Be meticulous.**

- [ ] Query `mart_ab_experiment` from DuckDB

**Pre-analysis:**
- [ ] Power analysis function:
  - Inputs: baseline conversion rate, MDE, alpha (0.05), power (0.80)
  - Output: required sample size per variant
  - Verify your synthetic experiment hit the required n
- [ ] Check sample ratio mismatch (SRM):
  - Expected 50/50 split — run chi-square test on assignment counts
  - Flag and document if SRM detected (even if synthetic — shows you know to check)
- [ ] Plot assignment volume over time — check for unexpected ramps or drops

**Primary analysis (Experiment 1 — conversion rate):**
- [ ] Frequentist: two-proportion z-test
  - Calculate observed lift, p-value, 95% CI on lift
  - State clearly: reject/fail to reject H0
- [ ] Bayesian alternative:
  - Beta-Binomial model (Beta prior + Binomial likelihood)
  - Calculate P(treatment > control), expected loss, credible interval
  - Plot posterior distributions of conversion rate for both variants
- [ ] Practical significance check: is the observed lift above your MDE threshold?
- [ ] Novelty effect check: plot daily conversion rate by variant over experiment window; flag if treatment effect decays

**Primary analysis (Experiment 2 — revenue per user, continuous metric):**
- [ ] Frequentist: Welch's t-test (unequal variance)
- [ ] Check normality assumption — revenue is typically skewed. Apply log transform or use Mann-Whitney U as robustness check
- [ ] Bootstrap CI on mean difference (1000 iterations minimum)
- [ ] Bayesian alternative: Normal-Normal conjugate or MCMC via PyMC/Stan

**Secondary / guardrail metrics:**
- [ ] Check that treatment didn't harm session count, review score, or refund rate
- [ ] Document all guardrail results regardless of significance

**Decision documentation:**
- [ ] Write a structured decision memo (Markdown file):
  - Experiment summary, metrics, results (both frequentist and Bayesian), guardrail checks, recommendation, caveats
  - This is what you show interviewers — the thought process, not just the p-value

- [ ] Save all outputs to `data/outputs/ab_testing/`

## 9. Analytics Module 3 — LTV Modeling

**Two-stage approach:**

**Stage 1 — Cohort LTV (simple, explainable):**
- [ ] Query `mart_cohort_retention` and `mart_customer_cohorts`
- [ ] Calculate ARPU (average revenue per active user) per cohort-period
- [ ] Project forward: LTV(t) = Σ (retention_rate(τ) × ARPU) for τ = 1 to t
- [ ] Plot LTV curves by cohort — identify which cohorts have highest projected LTV
- [ ] Calculate payback period: at what period does LTV exceed hypothetical CAC?

**Stage 2 — Probabilistic LTV (BG/NBD model):**
- [ ] Install `lifetimes` Python library
- [ ] Prepare summary data: `frequency`, `recency`, `T` (customer age), `monetary_value` per customer
  - Use `lifetimes.utils.summary_data_from_transaction_data()`
- [ ] Fit BG/NBD model (predicts future transaction frequency)
- [ ] Fit Gamma-Gamma model (predicts average order value conditional on being alive)
- [ ] Calculate predicted 12-month LTV per customer: `ggf.customer_lifetime_value(bgf, ...)`
- [ ] Validate: plot actual vs predicted purchases on holdout period (train on first 80% of timeline)
- [ ] Segment customers into LTV tiers (top 10%, mid 40%, low 50%) — use these as features in churn model

- [ ] Save model objects and predictions to `data/outputs/ltv/`

## 11. Data Science Module 2 — Churn Prediction

**Define churn:** customer with no order in last 90 days (adjust based on median inter-purchase time in your data — calculate this first).

- [ ] Build feature table per customer:
  - From `mart_rfm`: recency, frequency, monetary
  - From `mart_ltv_inputs`: avg order value, tenure, days between orders
  - From `mart_cohort_retention`: cohort-level retention rate (group feature)
  - From segmentation: segment label (one-hot or ordinal)
  - From `stg_order_reviews`: avg review score
  - LTV tier from Module 3
- [ ] Create binary target: `churned` = 1 if no order in last 90 days
- [ ] Check class balance — likely imbalanced. Document ratio.
- [ ] Train/test split: 80/20, stratified on target
- [ ] Baseline model: logistic regression (interpretable, interviewers love this as a starting point)
- [ ] Main model: XGBoost or LightGBM
  - Hyperparameter tuning: RandomizedSearchCV or Optuna
  - Handle imbalance: `scale_pos_weight` parameter or SMOTE on training set only
- [ ] Evaluate:
  - AUC-ROC (discrimination)
  - **Calibration curve** (critical — is predicted probability 0.7 actually 70% likely to churn?)
  - Brier score
  - Precision-Recall curve (more informative than ROC for imbalanced)
- [ ] If calibration is poor: apply Platt scaling or isotonic regression
- [ ] Feature importance: SHAP values (not just feature_importances_ — SHAP shows direction)
  - Plot SHAP summary plot and top 10 features
  - Be prepared to explain: "recency is the most important predictor, which aligns with the BG/NBD model assumptions"
- [ ] Connect back to LTV: calculate expected revenue saved per churner correctly identified

- [ ] Save model, predictions, and SHAP plots to `data/outputs/churn/`

## 12. Data Science Module 3 — Uplift Modeling

**This is your differentiator. Takes the A/B test further.**

**Goal:** Identify which customers would *respond* to a retention intervention (e.g., discount email), not just who is likely to churn.

- [ ] Use your A/B experiment data (Experiment 2 — discount campaign)
- [ ] Build feature table: same features as churn model per user
- [ ] Define target: `converted` (made a purchase in 30 days post-experiment)

**T-Learner (simplest, start here):**
- [ ] Fit model_control on control group users (features → outcome)
- [ ] Fit model_treatment on treatment group users (features → outcome)
- [ ] Uplift score per user = model_treatment.predict(X) - model_control.predict(X)
- [ ] Plot uplift score distribution — identify "persuadables" (high uplift), "sure things" (convert regardless), "sleeping dogs" (negative uplift)

**Qini curve / uplift curve:**
- [ ] Sort users by descending uplift score
- [ ] Plot cumulative incremental conversions vs population fraction
- [ ] Calculate Qini coefficient (analogous to AUC for uplift)
- [ ] Compare to random targeting baseline

**X-Learner (more robust with imbalanced treatment/control):**
- [ ] Implement using `causalml` library (`XLearner` class)
- [ ] Compare Qini coefficient to T-Learner

**Business framing:**
- [ ] Calculate: if we target top 20% by uplift score vs random 20%, what is the incremental revenue difference?
- [ ] Document: "sleeping dogs" — users who convert less when targeted. Flag these — avoid targeting them.

- [ ] Save uplift scores and Qini plots to `data/outputs/uplift/`
